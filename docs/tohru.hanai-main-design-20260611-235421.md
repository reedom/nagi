## Problem Statement

The builder wants an always-on AI concierge: a long-running service that awaits
requests from an incoming channel (Slack first), dispatches each request to the
right shape of agent work (one agent, a pipeline, or a swarm) via
ai-workflow-engine, replies in the originating Slack thread, and loops forever.
Today the engine is a one-shot CLI (`ai-workflow-engine run`); there is no
inbound channel, no dispatcher, and no daemon shape. The concierge will live in
a NEW repo and consume the engine as a library.

## What Makes This Cool

Message the bot from a phone, and a deterministic multi-agent workflow runs on
the builder's own machine â€” local agent CLIs (claude, codex), real budgets,
real pipelines â€” with the result landing back in the same Slack thread. The
chosen "smart dispatcher" framing means the concierge classifies each request
and picks the right workflow, not just a fixed slash-command menu. The
escalation broker runs in reverse: mid-run approval requests surface in the
same thread and a human reply unblocks the agents. None of the popular
chat-agent platforms dispatch to deterministic JS-orchestrated local-CLI
swarms; that combination is the novelty.

## Constraints

- Solo builder, personal/open-source tool; optimizes for daily usefulness soonest.
- New repo; ai-workflow-engine stays a clean runtime library (TypeScript, pnpm, vitest).
- No Slack bridge exists today; the front door must be built (socket-mode bot).
- Agentbus today is a unix-socket escalation broker only (engine â†’ human), not a message bus.

## Premises

1. The concierge lives in a NEW repo and uses ai-workflow-engine as a library.
   The engine's programmatic API already exists â€” `runWorkflow`/`loadWorkflow`
   in `src/runtime/runner.ts` accept injected adapters, args, budget, and an
   escalation `ApprovalChannel`, and return a result with no process-exit
   semantics. The engine repo only needs library `exports` in `package.json`
   (currently `bin`-only) re-exporting the runner, adapter factories, and
   `ApprovalChannel` types.
2. (Revised by builder) Build our own thin Slack socket-mode bot in the
   concierge repo â€” no OpenClaw/nanobot dependency. Study OpenClaw's UX and
   threat model, but own the front door.
3. v1 dispatch is a hand-registered workflow registry plus an LLM triage agent
   that picks a workflow and extracts args. Compose-workflow-on-the-fly is
   explicitly v2.
4. The existing escalation broker is reused so running workflows can ask the
   human questions back in the same Slack thread. The round-trip is v1 scope
   because it is what makes async dispatch trustworthy.

## Approaches Considered

### Approach A: Thin Slice (minimal viable) â€” CHOSEN
Single Node process in the new repo: Bolt socket-mode listener â†’ triage
`agent()` call â†’ registry lookup â†’ engine `runWorkflow` â†’ reply in thread.
One request at a time, in-memory state, escalation answers read from thread
replies. Effort Sâ€“M (a weekend with Claude Code). Risk low.
Reuses: `runWorkflow`/`loadWorkflow` from `src/runtime/runner.ts`, both
adapters, the agentbus escalation channel as-is.
Cons: no persistent queue â€” in-flight and queued work is lost on crash;
requires adding package exports to the engine (small â€” the runner API itself
already exists).

### Approach B: Concierge Daemon (ideal architecture)
Persistent request queue (SQLite/JSONL journal), N-concurrent runs with
per-request budgets, progress events streamed to threads, crash recovery, and
a first-class engine embed API (`createEngine({adapters, escalation})`). The
escalation seam needs no generalizing â€” `ApprovalChannel` in
`src/escalation/types.ts` is already channel-generic. Effort L (~1â€“2 weeks of
sessions).
Risk medium: real persistence/concurrency/recovery complexity before first
daily value. This is the documented v1.5 hardening path, not v1.

### Approach C: Foundry-First (creative/lateral)
Approach A's slice, but the registry is the product: workflows are
frontmatter+JS files in `workflows/`, hot-reloaded; the triage "no match" path
drafts a new workflow file, opens a PR, and asks in-thread "want me to learn
this?" Keeps premise 3 intact because new capability only lands through
reviewed PRs. Effort Mâ€“L (~1 week). Risk medium: generation quality unproven;
PR automation built before the basic loop is solid. This is the documented v2
north star.

A proves the full loop (Slack in â†’ triage â†’ workflow swarm â†’ escalation

v1 component sketch:
- **Engine repo changes (two, both small):**
  - Add library `exports` to `package.json` re-exporting `runWorkflow`/
    `loadWorkflow`, the adapter factories, and `ApprovalChannel` types â€” the
    runner API itself already exists.
  - Thread a `cwd` through `RunOptions`: `startEscalation` currently resolves
    permission defer rules via `loadSettingsDeferRules(process.cwd())`, which
    in a daemon would read the concierge's cwd instead of the target repo's.
- **Concierge repo:**
  - `slack/`: @slack/bolt socket-mode app; app-mention/DM â†’ `RequestContext`
    {channel, threadTs, text, user}. **Authorization is a v1 requirement, not
    an option:** an allowlist of Slack user IDs gates dispatch, because the
    claude adapter grants unrestricted Bash on the host machine.
  - Thread state: an in-memory map of `threadTs â†’ pending interaction`
    (clarification questions only â€” escalation approvals use Block Kit buttons
    and never consume thread replies; eng review D11). A thread reply is
    consumed by its pending clarification if one exists; otherwise it is
    treated as a new request. A clarification reply re-runs triage with the
    original request plus the reply appended. Entries carry a TTL
    (check-on-access plus periodic sweep; eng review D9/8A): an expired
    clarification means a late reply is treated as a fresh request.
  - `registry/`: workflow entries `{id, description, argsSchema (zod), run}`.
  - `triage/`: a direct single-call to the claude adapter (not a registered
    workflow â€” `agent()` exists only inside workflow bodies) returning
    `{workflowId, args, confidence, clarificationQuestion?}`; below the
    confidence threshold, post the clarification question in-thread instead of
    running.
  - `escalation/`: a Slack-thread `ApprovalChannel` impl â€” approval requests
    posted in the thread as Block Kit messages with Approve/Deny buttons
    carrying the request id (eng review D11; misrouted approvals become
    structurally impossible; tool input shown in a code block, long payloads
    attached as a snippet, never silently truncated). Concurrent escalations
    from a parallel swarm are serialized inside the channel (internal FIFO;
    eng review D2/1A). Policy: `onTimeout: 'wait'` â€” note the claude adapter
    caps 'wait' at a 24-hour hook timeout (claude.ts hookTimeoutSeconds), so
    "wait" means "up to the adapter ceiling", not indefinite.
  - Seed workflows: "review this repo/diff and summarize risks" plus one
    research pipeline.
  - Single-flight policy for v1: a second request gets an immediate "busy with
    [current task], queued" reply and runs when the current one finishes
    (simple in-memory FIFO, no persistence).

## Open Questions

- Confidence threshold and triage model choice (cheap/fast vs. accurate) â€” tune
  with real traffic.
- Per-workflow permission rules beyond the user allowlist
  (`src/escalation/rules.ts` is the seam to reuse).
- Engine API surface: minimal function exports vs. full `createEngine` factory
  (defer the factory to B unless A needs it).
- Escalation timeout ceiling: `onTimeout: 'wait'` indefinitely vs. a multi-hour
  cap with a reminder ping in-thread.

## Success Criteria

- From a phone: send a Slack message, get a triage acknowledgment in-thread
  within seconds, and a workflow result in the same thread.
- A workflow that hits an uncovered tool call posts an approval request in the
  thread, and a human reply unblocks it (full agentbus round-trip).
- An unknown request gets a clarifying question, not a silent failure.
- A failed run (workflow throw, budget exhausted, adapter crash) posts the
  error summary in-thread â€” never silence.
- Messages from non-allowlisted Slack users are politely refused and never
  reach triage or the engine.
- The engine repo gains only package exports and a `cwd` run option;
  `pnpm test` stays green in both repos.

## Engineering Review Decisions (2026-06-12, /plan-eng-review)

Review findings (sections 1-4):
1. (1A) SlackApprovalChannel serializes concurrent escalation requests
   internally â€” one approval question in a thread at a time. Concurrent
   `channel.request()` calls are real: broker.ts:67 has no queue and
   `parallel()` launches agents eagerly.
2. (2A, confirmed twice) Registry entries embed a real `WorkflowModule`
   (`{meta, default(api)}`); the concierge calls
   `runWorkflow(entry.module, ...)`. Chosen explicitly for foundry symmetry:
   generated files and registry entries share the engine's one workflow format.
3. (3A) Default per-request token budget in concierge config, per-entry
   override. Engine support exists end to end; v1 just passes a number.
4. (4A) Triage args failing the entry's zod schema are treated as a
   clarification (question names the bad field; reply re-runs triage) â€” one
   unified "can't dispatch yet" path. A triage-returned workflowId not in the
   registry is handled the same way.
5. (5A) The dispatch acknowledgment echoes the chosen workflow and extracted
   args; dispatch proceeds immediately (no cancel window).
6. (6A) Crash recovery is fail-fast + launchd `KeepAlive` â€” no in-process
   Bolt reconnect handling (upstream socket-mode reconnect bugs: bolt-js
   #1151, node-slack-sdk #1495). In-memory queue loss on restart is accepted
   for v1 and documented.
7. (7A) Triage eval suite ships in v1: 15-20 golden cases (one per registry
   entry + clarification triggers + hallucinated-id), grown from dogfood
   misfires.
8. (8A) Thread-state entries carry a TTL with check-on-access + sweep;
   escalation lifetimes are owned by the broker policy.

Outside-voice (Codex) tensions, all accepted:
9. (D10) Engine: `RunOptions.cwd` becomes the run-level DEFAULT agent cwd in
   `createWorkflowApi` (per-call override wins) AND the defer-rules cwd â€” one
   cwd concept per run. Regression tests pin both "no cwd â†’ process.cwd()"
   and "cwd set â†’ agents spawn there".
10. (D11) Escalation approvals use Block Kit Approve/Deny buttons bound to a
    request id. Deletes the approve/deny text parser and the nudge state.
11. (D12) Control commands `status` and `cancel` bypass the workflow queue
    and are handled immediately. `cancel` kills the active run's child
    process group, closes the broker, posts cancellation in the run's thread.
12. (D13) Repo references are a configured alias map (name â†’ absolute path);
    workflow arg schemas take an enum of known aliases â€” free-form paths are

13. (D14) Append-only JSONL audit log (timestamp, team/user, raw text, chosen
    workflow, args, approvals, token usage, outcome; best-effort writes that
    never crash the request path) + the auth gate pins workspace/team ID
